# Provenance & independence — keeping AROS uncontaminated

> Drafted 2026-06-24 · Applies to every `spec.md` under `docs/features/*/`

## Why this exists

AROS is **APL / LGPL-2.1**, and we keep it that way. We build this bridge from
**public APIs and published standards**, the **AROS tree** (ours to use), and **our
own spikes** — and from nothing else. We do **not** read, search, or consult any
third-party *implementation* source — emulator, guest agent, driver, or otherwise —
in producing this work, and **any resemblance to existing implementations is
coincidental**. That posture is what protects AROS's licence: there is no foreign
code to import, paraphrase, or structurally mirror, because we never look at any.
This doc records the discipline that keeps it true.

> Not legal advice. This is an engineering process to minimise copyright risk; the
> project owner confirms the licence strategy before anything ships.

## The two roles

We keep a lightweight author/implementer split — but **both work only from approved
sources**: public APIs and published standards, the AROS tree (APL/LGPL — ours), and
this project's own spikes. **Neither reads any third-party implementation source.**

**Role A — Spec Author.** Works from the approved sources above to produce a
**functional specification**: interface contracts, required observable behaviour,
algorithms described as *requirements in our own words*, and acceptance tests. Records
**provenance** for every requirement. Writes the `spec.md`.

**Role B — Implementer.** Implements **solely** from (1) Role A's `spec.md` and (2) the
approved public sources below. Produces the AROS-licensed code. *(A fresh agent.)*

The split is for clarity and review, not a firewall around foreign code — there is no
foreign code in the loop. A's spec carries *facts and contracts*, never the *expression*
of any third-party implementation, because none is ever read.

## What a spec MAY contain

- **External interface signatures** — Apple framework APIs (AppKit/Metal/CoreAudio/…),
  AROS in-tree headers (APL/LGPL — already ours to use), POSIX. These are interfaces,
  not the GPL code's expression.
- **Data layouts dictated by an external standard** — IFF FTXT, AmigaOS hunk, `fd_set`,
  the AHI sub-driver struct. Mandated by the format/API, not authored by any third party.
- **Algorithms as requirements**, in our own prose or neutral pseudocode — *what* must
  happen and *why*, derived from an independent constraint.
- **Acceptance tests** — the unattended-loop markers and pixel/byte/RMS asserts.
- **Provenance tags** (below) on every non-trivial requirement.

## What a spec MUST NOT contain

- Verbatim or paraphrased third-party implementation source, or its comments.
- A third-party implementation's identifier names, call sequence, or file/function
  layout used as a template. (Names dictated by an *external* API — `nextDrawable`,
  `WaitSelect` — are fine; names that are some implementation author's invention are not.)
- Any requirement whose *only* justification is "because some implementation does it
  that way." If a behaviour can't be re-derived from a public source or our own
  constraints, it doesn't belong in the spec.

## Provenance tags (use them in every spec)

Tag each substantive requirement so Role B knows its footing:

- **`[PUB]`** — public API / published standard (Apple docs, POSIX, the IFF spec). Free
  to implement directly.
- **`[AROS]`** — an in-tree AROS header/driver (APL/LGPL). Cite the path; ours to use.
- **`[OURS]`** — this project's own spikes (`hosted/*`, `graft/*`, the H-series).
- **`[DERIVED]`** — an *independently-derived* requirement, flagged for extra
  verification. It stands solely on its cited `[PUB]`/`[AROS]`/`[OURS]` justification;
  the flag just marks it as one we reasoned out ourselves and therefore want
  double-checked. Role B implements from that justification. If you can't attach an
  independent justification, cut it.

Role B may rely on `[PUB]`/`[AROS]`/`[OURS]` freely. `[DERIVED]` items must each
stand on their independent justification alone.

## Licence map (what's off-limits vs. adoptable)

| Source | Licence | How we may use it |
|--------|---------|-------------------|
| GPL Amiga emulators | GPL-2.0 | **Not read or used — independent work.** No source consulted; any resemblance is coincidental. |
| **Emu68** | **MPL-2.0** | **Adoptable as isolated files** (file-level copyleft): keep its files verbatim with their MPL header in a quarantined dir, don't intermix with AROS source. Clean-room *not* required, but a clean integration boundary is. See `68k-jit/design.md` `[J0]`. |
| AROS tree | APL / LGPL-2.1 | Ours. Cite and build on directly. |
| Apple frameworks | proprietary, public API | Link against; implement to the published interface. |
| **vbcc / vasm / vlink** (Barthelmann/Wille) | free for non-commercial **+ explicit commercial exception for M68k/AmigaOS targets** (vbcc/vasm); vlink freeware | **Build TOOLS, not emulators — not GPL/MPL emulator source.** Built from source on this Mac (`hosted/jit68k/apps68k/tools/{build-vasm,build-vbcc}.sh`, binaries gitignored under `.toolchain/`), used to COMPILE the `[J5m]` (integer) and `[J5t]` (HARDWARE-FP, `-fpu=68881`, dtgen cross=y for big-endian FP constants) C test programs into 68k AmigaOS hunk executables (`vbcc → vasm → vlink`). No emulator/decoder source is read or copied; these are a C compiler + assembler + linker. License text captured in `.toolchain/{VBCC,VLINK}-LICENSE.txt`. |
