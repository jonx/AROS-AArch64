## 2. THE IN-FLIGHT BUG — diag stack-scanner faults on NULL reg-frame

> **FIXED 2026-07-18** — aros-upstream `816da226` ("unix/kernel: make the
> SIGINFO diag scanner fallible instead of faulting"). Implemented as
> sketched below plus range-aware validation everywhere (whole structure
> must sit in one exec region, not just the first word) and guarded list
> walks in the handler. Validated per the recipe: 89 SIGINFO dumps fired
> into heavy scroll/nav paint traffic; guards skipped 90 unreadable
> frames (each a pre-fix fault candidate); zero traps, app alive and
> navigating afterwards. Pre-fix the same profile crashed in ~10 s.
> Section kept for the diagnosis record.

**Symptom:** app runs ~10s under real use, then Guru
`Module kernel .text+0x1B3C` (offset moves with kernel edits; it was
+0x1AA0 in older kernels — same function), task shows as `C:Feraille` or
`pthread thread #N`, DEADEND. Instance survives (containment).

**Diagnosis (complete, high confidence):** the faulting code is
`core_DiagSemCandidates` in `arch/all-unix/kernel/kernel.c` — the
crash-containment branch's diagnostic that scans a task's saved registers
and stack words for semaphore candidates. It runs from `core_DiagHandler`
(SIGINFO), and cocoametal's watchdog (`cm__host_tick`) fires SIGINFO
automatically whenever the guest looks stale — which now happens during
perfectly legitimate long tiny_skia paint bursts. The scanner then walks
`regs->x[19..28]` — and for tasks whose `regs` / `et_RegFrame` is NULL or
stale, that's a dereference through ~NULL (measured fault addr 0x110,
x20=0x108 = `&regs->x[i]` with regs≈NULL; disassembly at +0x1B3C matches
the scanner's `mov w8,#0xf; ldr x22,[x20,#8]; ldr x0,[x23]` +
3-arg-call pattern of `core_DiagSemReport(tag, num, ss)`).

Confirmed chain: SIGINFO interrupts `tiny_skia::pipeline::lowp::start`
(render) → `_sigtramp` → diag handler → scanner faults → the *diagnostic*
crash deadends the interrupted task. Every historical
"`.text+0x1AA0` Software Failure" was this same scanner, i.e. the
watchdog converting freezes/stalls into Gurus.

**Fix (sketched, not yet applied):** in `arch/all-unix/kernel/kernel.c`:

- `core_DiagSemCandidates(t, regs)`: bail out unless
  `core_DiagValidPtr(regs)`; also range-check `regs->sp` reads (already
  partly done) — the register loop is the unguarded part.
- `core_DiagTask(t, NULL)` path: it already checks
  `tc_ETask && et_RegFrame` before using the saved frame, but the frame
  *contents* can be stale/garbage for a never-yet-switched task — wrap
  the whole per-task diag in the same validity checks.
- `core_DiagBacktrace(pc, fp)`: validate `fp` chain entries with
  `core_DiagValidPtr` before dereferencing (it has a bound + alignment
  check; add the mem-range check).
- Optional belt: rate-limit or make the SIGINFO scanner opt-in
  (`AROS_DIAG_SEMSCAN=1`), since it dereferences guesses BY DESIGN.

**Verify:** rebuild kernel (§4 targets!), then real-mouse use for a few
minutes; also `kill -INFO <Macaros pid>` repeatedly DURING heavy paints
(navigate a big dir and immediately dump) — pre-fix that reproduces the
scanner crash, post-fix it must print dumps and never trap.

## 3. What was fixed (all committed; per-repo map in §6)

### 3.1 ROOT CAUSE — hosted darwin never preempted CPU-bound tasks
`aros-upstream 8f1acacc` + full analysis in `graft/UPSTREAM-NOTES.md`
item 38. Preemption ticks (process-directed SIGALRM from ITIMER_REAL)
were delivered by darwin to arbitrary host threads; `core_IRQ` dropped
off-thread ticks. Measured with the T-TICKPROBE counters (printed in
every SIGINFO task dump): under a spin, **0** ticks reached the AROS
thread in 10s, ~237/s dropped. A task that never blocked therefore ran
forever: UI starved (task dumps showed `C:Feraille` READY with its wakeup
signal already delivered), timer.device starved, guest software clock
frozen for minutes of wall time. Fix: forward mis-delivered ticks
thread-directed via `pthread_kill(aros_host_thread, sig)`. After: 4293
handled / 35 forwarded; all ClockTest variants pass; the scroll stress
that froze every run holds ~3-10% CPU.

### 3.2 pthread (aros-upstream `cacf6ddc`, earlier commits)
- `sched_yield` did two `SetTaskPri()` (ready-list re-sorts) per yield —
  now the AttnResched-flag reschedule (yields are ~ns now).
- `pthread_cond_timedwait` opened/closed timer.device on EVERY call
  (T-TIMERSHARE: one shared open, cloned per request).
- Expired/negative deadlines passed `timerisset()` (true for negative!)
  and sent timer.device negative tr_time; fixed — and the fix exposed:
- **Early-return teardown bug**: returning ETIMEDOUT through
  `CloseTimerDevice()` with a request that was never SendIO'd →
  `AbortIO` → `Remove()` through uninitialized/NULL `ln_Pred` → the
  `Exec_42_Remove` Gurus. Fixed by validating the deadline BEFORE
  opening the timer; plus a T-ABORTGUARD log-and-skip in
  `CloseTimerDevice`.
- exec `Alert()` and `ShutdownA()` now log unconditional breadcrumbs
  (deadend Gurus + ColdReboot re-exec used to destroy their own
  evidence).

### 3.3 posixc (aros-upstream, committed 2026-07-16)
Unlocked shared fd table (`__fdesc.c` T-FDLOCK): semaphore-guarded table
+ pool, atomic find+claim in open/opendir/pipe/dup/dup2/fcntl. Proven by
A/B: reverting it produced deterministic `AddTail` bus faults.

### 3.4 gpui_aros (zed-aros `aros-platform`, four commits)
- Run loop blocks on exec `Wait()` + one long-lived timerequest instead
  of polling `usleep(2000)` 500×/s (idle CPU ~100% → ~1.4%).
- Workers + timer thread run at pri -1 (below UI; equal-pri workers
  starved the UI on the single guest CPU).
- Worker pool capped at 2 on AROS (host core count is meaningless on a
  1-CPU guest).
- Dispatcher queues are a std Mutex/Condvar blocking queue — crossbeam
  AND parking_lot spin before parking (parking_lot's generic
  ThreadParker busy-spins on this target: measured 99% CPU at idle).
- `gpa_task_ptr()` diagnostic (maps task-dump pointers to thread roles).

### 3.5 Feraille (main)
- `obs.rs`: crash reports persist to `MacRW:feraille-panic.txt`
  (essential facts written FIRST, before backtrace capture);
  non-panicking stderr writes (`eprintln!` panics on stderr failure —
  with panic=abort one lost log line rebooted the whole OS); AROS log
  bridge capped at Info (`FERAILLE_LOG_TRACE=1` for the firehose).
- Folder-size walker re-gated on AROS (`folder_sizes.rs`).

### 3.6 Harness (aros-aarch64 main)
- `hosted/timertest`, `hosted/clocktest` — the on-device tests that
  cracked the case. `C:ClockTest pure` is the 10-second regression test
  for preemption forever.
- `rebuild-aros.sh` fails loudly on mmake's "Nothing known about target"
  (see foot-guns).
- `aros-ctl` crash-grep no longer false-positives on task dumps.
- `aros-host-conf.sh`: explicit env beats the conf file
  (`AROS_HOST_MEMORY` was silently clobbered to 256 MB → Rust OOM
  panics misread as spontaneous reboots). Conf now says `memory 1280`.

## 4. Build & test playbook (with every foot-gun we hit)

Build loop per layer:

| change in | build with | then |
| --- | --- | --- |
| Feraille / gpui_aros (Rust) | `cargo +nightly-2026-06-27 build --release -p feraille-aros-app --target ../aros-aarch64/hosted/rust/aarch64-unknown-aros.json -Zjson-target-spec -Zbuild-std=std,panic_abort` | `PROFILE=release crates/feraille-aros-app/link-aros.sh` |
| pthread | `TARGETS="linklibs-pthread" graft/rebuild-aros.sh` | relink Feraille (static lib!) |
| posixc | `TARGETS="compiler-posixc" graft/rebuild-aros.sh` | nothing (runtime .library) |
| kernel | `TARGETS="kernel-kernel kernel-link-unix boot" graft/rebuild-aros.sh` | nothing |

Run/drive: `AROS_CTL_STARTUP_FILE=.../feraille.startup graft/aros-ctl run`,
then `aros-ctl click/wheel/menu/shot/log/status`.

**Foot-guns (each cost us real time):**

1. **mmake exits 0 on unknown targets.** `compiler-pthread` never
   existed (real name `linklibs-pthread`); two crash fixes "failed
   validation" because the OLD library kept shipping. rebuild-aros.sh
   now detects it, but ALWAYS verify a deploy by grepping a marker
   string from your change in the artifact:
   `grep -c MYMARKER .../AROS/C/Feraille`.
2. **Kernel = packaged kickstart.** `kernel-kernel` alone rebuilds only
   `Devs/kernel.resource`; the bootstrap loads `boot/darwin/kernel`
   which `kernel-link-unix` links (then `boot` stages). Copying
   kernel.resource over it bricks the boot ("Failed to create
   ExecBase"). Stale `mmake.cache` in `~/aros-build` causes phantom
   no-ops — delete it when in doubt.
3. **`grep` is aliased to ugrep** on this box and silently missed
   matches more than once (e.g. `ARF_AttnSwitch` in a public header).
   Use `/usr/bin/grep` in anything that matters.
4. **`aros-ctl diag` KILLS the instance.** On a live freeze use
   `kill -INFO <Macaros pid>` — non-destructive task dump into
   `/tmp/aros-window.log` (now includes the T-TICKPROBE tick counters).
5. **`crash=none` / `state=running` is NOT liveness.** Livelocks and
   contained Gurus both report clean. Real signals: app-log growth
   (`~/AROS/Shared/feraille-log.txt`), per-thread CPU
   (`ps -M <pid>` — one thread at ~100% = a spin), and the guest clock
   advancing (log timestamps vs wall clock).
6. **Synthetic input ≠ real input.** `aros-ctl click` teleports; wheel
   events reproduce CPU storms, but several bugs only fell to a human
   at the mouse. Trust the human report over a green harness.
7. **A guest "reboot" is the bootstrap re-exec'ing** (child exit 0x81 →
   `Host_ColdBoot` → execvp): new pid, truncated host log. Capture logs
   truncation-proof: `tail -n +1 -F /tmp/aros-window.log > sidecar &`.
8. Boot needs `AROS_HOST_MEMORY` ≥ 1280 (now the conf default) and the
   ~20s `feraille.startup` (Stack 16MB + SetEnv HOME).

## 5. Diagnostic toolkit (all permanent now)

- `C:ClockTest [pure|clock|yield|cond]` — does a pri -1 spinner wedge a
  pri 0 timer wait? `pure` must reach `[D]`. THE preemption regression
  test.
- `C:TimerTest` — do timed-wait primitives actually wait?
- T-TICKPROBE counters in every SIGINFO dump: `irq ticks:
  handled-on-aros-thread=N dropped-off-thread=M` — M exploding while N
  sits still = tick delivery broken again.
- exec ALERT/ShutdownA breadcrumbs in the host log.
- `MacRW:feraille-panic.txt` (≈ `~/AROS/Shared/`) — Rust panics.
- `gpa_task_ptr()` + kernel task dumps map "pthread thread #N" → role.
